IPFS ROLE CORRECTED ΓÇö IPFS stores only transaction metadata JSON (linked via CID in transaction_log). IPFS does NOT store submission files, bounty terms, or any other content 3. ESCROW CORRECTED ΓÇö funds held in smart contract application account only. No platform admin wallet. No external escrow account 4. Cloudflare R2 still handles submission files and avatars IPFS Exact Role in This System When any significant on-chain transaction occurs (bounty creation, submission, approval, rejection, dispute, payout, DAO vote, resolution), the backend creates a structured metadata JSON object describing that transaction. This JSON is pinned to IPFS via Pinata. The returned CID is stored in transaction_log.ipfs_metadata_cid. This gives every transaction a permanent, verifiable, human-readable audit record that cannot be altered. Escrow Account ΓÇö Exact Mechanism The Algorand smart contract application account IS the escrow. When a creator locks funds, they send ALGO directly to the contract app account address via a grouped transaction (Payment txn + create_bounty app call). The contract holds the ALGO. Only the contract&apos;s own ABI methods (approve_payout, reject_submission with counter=0, resolve_dao_dispute, refund_expired, let_go_bounty, cancel_bounty) can release funds via InnerTransactions. No backend wallet, no admin key, no external account involved. 1.
